Zero Trust Architecture on a Kubernetes Home Lab

A progressive, hands-on tutorial that walks the ZTA-2026-001 ARB submission — a NIST SP 800-207-aligned architecture for a global bank — into a single-node Docker Desktop Kubernetes cluster. Seven labs, one per 800-207 tenet, each with runnable manifests, validation commands, and a break-it exercise.

EnvironmentDocker Desktop · Kubernetes 1.29+
AudienceIntermediate K8s practitioners, ZTA-new
Runtime45–90 min per lab
FrameworkNIST SP 800-207 (Aug 2020)

1 · Executive Overview

The source document, ZTA-2026-001, is a C4 × NIST SP 800-207 architecture prepared by a global bank's Security Solutions Architecture team. It proposes a distributed Policy Engine (OPA/Cedar), a policy-as-code Policy Administrator (Styra DAS + GitOps), and a heterogeneous fleet of Policy Enforcement Points (Kong FAPI, Istio, Zscaler ZPA, CyberArk, Broadcom AAM, Netskope, an in-house AI Gateway). Workload identity is SPIFFE/SPIRE; workforce identity is Microsoft Entra ID federated with Okta; customers sit on ForgeRock CIAM with FAPI 2.0 and DPoP-bound tokens. Signals flow from CDM (Wiz, Tanium), threat intel, SIEM (Splunk + Snowflake), UEBA, fraud/AML (Featurespace) and data-access policy (Immuta) into a Trust Algorithm that emits a continuous trust_score. This tutorial reconstructs the essential shape of that architecture on a laptop — using only open-source projects — so that each 800-207 tenet becomes a thing the reader can deploy, break, observe, and repair.

2 · Lab Environment Setup

2.1 Prerequisites checklist

  • Docker Desktop 4.30+ (macOS, Windows with WSL2, or Linux) with built-in Kubernetes enabled. Settings → Kubernetes → Enable Kubernetes.
  • Resources: at minimum 6 CPU / 8 GB RAM / 60 GB disk. Istio + SPIRE + Falco + Keycloak + Grafana stack is chatty. 10 GB is more comfortable.
  • CLI tools: kubectl ≥ 1.29, helm ≥ 3.14, kustomize ≥ 5.0, istioctl ≥ 1.22, jq, yq, openssl.
  • Kubernetes context: docker-desktop. Every command in this tutorial is prefixed with --context docker-desktop or assumes you have set it as the current context.
  • Host entries for ingress: add 127.0.0.1 bookstore.local keycloak.local grafana.local to /etc/hosts (macOS/Linux) or C:\Windows\System32\drivers\etc\hosts (Windows).
Docker Desktop caveats

LoadBalancer: Docker Desktop's built-in Kubernetes assigns localhost as the external IP for Service type LoadBalancer — this is usable, but ports must be free. Cilium: kernel-level features (eBPF bandwidth manager, L7 policies) are limited under Docker Desktop's Linux-in-VM kernel; we therefore use Istio as the primary mesh and mention Cilium only in the extensions. Falco: we use the modern_ebpf driver, which works on Docker Desktop 4.30+ without kernel module compilation.

2.4 Bootstrap — cert-manager, Istio, SPIRE

cert-manager provides a cluster-wide CA for non-SPIFFE TLS (Keycloak ingress, Grafana). Istio is the east-west PEP and sidecar mTLS enforcer. SPIRE issues short-lived X.509 and JWT SVIDs for workload identity — the 800-207 §3.1.3 ID Management component in this lab.

Bashbootstrap/01-cert-manager.sh
#!/usr/bin/env bash
set -euo pipefail
CTX=docker-desktop

helm repo add jetstack https://charts.jetstack.io
helm repo update

helm --kube-context $CTX upgrade --install cert-manager jetstack/cert-manager \
  --namespace cert-manager \
  --version v1.15.3 \
  --set crds.enabled=true \
  --set global.leaderElection.namespace=cert-manager \
  --wait --timeout 5m

kubectl --context $CTX -n cert-manager wait --for=condition=Available \
  deploy/cert-manager deploy/cert-manager-webhook deploy/cert-manager-cainjector \
  --timeout=180s

# Cluster-wide self-signed issuer (non-SPIFFE paths only)
cat <<'EOF' | kubectl --context $CTX apply -f -
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: zta-selfsigned
spec:
  selfSigned: {}
EOF
echo "cert-manager ready."
Bashbootstrap/02-istio.sh
#!/usr/bin/env bash
set -euo pipefail
CTX=docker-desktop
ISTIO_VERSION=1.22.3

# Install via istioctl with the 'default' profile — adequate for the lab.
# Sidecar mode (not ambient) for clearest pedagogical mapping to PEP.
istioctl --context=$CTX install -y \
  --set profile=default \
  --set values.global.proxy.resources.requests.cpu=10m \
  --set values.global.proxy.resources.requests.memory=64Mi \
  --set meshConfig.accessLogFile=/dev/stdout \
  --set meshConfig.enableTracing=true \
  --set values.pilot.env.PILOT_ENABLE_AMBIENT=false

kubectl --context $CTX -n istio-system wait --for=condition=Available \
  deploy/istiod deploy/istio-ingressgateway --timeout=240s

# Mesh-wide mTLS defaults to PERMISSIVE out-of-box; Lab 2 flips to STRICT.
echo "Istio ${ISTIO_VERSION} installed; sidecar injection enabled on zta-* and bookstore-* namespaces."

All data sources and computing services are resources

NIST SP 800-207 §2.1 Tenet 1 — "An enterprise considers all of its data sources and computing services as resources."

Tenet
T1 Resources (§2.1)
Components
Enterprise Resource (§3.1); CDM Data Source (§3.2) seeded
Variant
All four deployment variants begin here — you cannot protect what you have not enumerated
Source §
§4 (C4 L2 — Enterprise Resources), §7 (Component catalogue), §8.8 (data classification)

Learning objectives

  • Enumerate every pod, service, and persistent volume in the cluster as a discrete resource per 800-207 §3.1.
  • Apply a consistent label schema (zta.resource, zta.data-class, zta.tier-role, zta.owner) that later labs will key off.
  • Produce a resource inventory usable as a CDM stand-in, mirroring the source §7.14 (Wiz + Tanium).
  • Recognise the source document's five-tier data classification (§8.8) and encode it as Kubernetes labels.
  • Distinguish subjects, resources, and assets — the three entities 800-207 §2 separates.

Concept primer

The source document catalogues ~20 named components (§7) and places them inside four logical layers (Data Sources, Control Plane, PEPs, Enterprise Resources) at C4 L2 (§4). NIST 800-207 calls all of these — databases, APIs, services, even agents — resources, and Tenet 1 begins by insisting that the enterprise inventory them all and treat them as equally worthy of protection irrespective of whether they run on z/OS, EKS, or a laptop. In this lab the cluster plays the role of the bank's estate at miniature scale: the bookstore's frontend, api, and db are the "Subjects · Assets · Resources" box in the source diagram; the Istio sidecars are not resources, they are PEPs; OPA is the PE. The label schema you apply below becomes the primary key for every later lab's policy — policies will select resources by label, not by name, so that adding a new workload automatically inherits the correct enforcement without policy edits. This is the 800-207 §3.4.1 Trust Algorithm input schema, rendered in Kubernetes-native form.

Lab 02 · 60 min

All communication is secured regardless of network location

NIST SP 800-207 §2.1 Tenet 2 — "All communication is secured regardless of network location."

Tenet
T2 Secured communications (§2.1)
Components
PEP (Envoy sidecar, §3.1.2), PKI Data Source (Istio CA; source §7.15 analogue)
Variant
Device-agent/gateway (§3.2.1) — each workload's sidecar is the on-resource gateway that terminates mTLS
Source §
§4 (Istio mesh + SPIFFE), §7.8 (Istio), §8.5 (telemetry), §8.7 (API security)

Learning objectives

  • Flip mesh-wide PeerAuthentication from the Istio default (PERMISSIVE) to STRICT mTLS.
  • Prove east-west encryption empirically with tcpdump inside a debug pod — not by trusting the control plane's word.
  • Observe that "inside the cluster" carries no trust — an in-cluster plaintext client is rejected.
  • Map the observed behaviour to source §4 and §8.7 and to 800-207 §2.1 T2 and §3.4.2 (implicit trust zone minimisation).

Concept primer

The source document's §3 Context warns that no network segment, including the corporate LAN, is trusted by default, and §4 places Istio at every east-west hop with SPIFFE SVIDs rotated every 60 minutes. NIST 800-207 §2.1 T2 is categorical: it does not matter whether traffic is "internal" or "external"; the same authentication and encryption obligations apply. The subtle point is that T2 does not by itself demand mTLS — it demands the strongest possible method — but in practice a modern mesh delivers mTLS cheaply enough that it is the default. What T2 rules out is the older defence-in-depth fallacy that perimeter firewalls can justify plaintext inside. The proof in this lab is deliberately adversarial: rather than reading the Istio control plane's claim that mTLS is on, you sniff a live packet from an untrusted-looking pod in an unrelated namespace. The bytes are opaque or not; there is no middle ground.

Step-by-step

01Apply the mesh-wide STRICT PeerAuthentication. Per-workload policies can still relax this, but nothing does by default.

Lab 03 · 75 min

Access to individual enterprise resources is granted on a per-session basis

NIST SP 800-207 §2.1 Tenet 3 — "Access to individual enterprise resources is granted on a per-session basis."

Tenet
T3 Per-session access (§2.1)
Components
ID Management (§3.1, Keycloak analogue of ForgeRock CIAM / Entra ID), PKI (§3.1, SPIRE analogue of Venafi + AWS PCA)
Variant
Resource-portal (§3.2.3) for human→API; device-agent/gateway (§3.2.1) for workload→workload
Source §
§1 (DPoP-bound tokens, 60-min SVIDs), §5 Flow A (5-min token), §7.1, §7.4, §8.2

Learning objectives

  • Configure Keycloak to issue 5-minute OAuth2 access tokens with a realm-specific audience.
  • Configure SPIRE to issue 5-minute X.509 SVIDs and observe rotation in real time.
  • Demonstrate that session expiry forces re-authentication — the defining behaviour of T3.
  • Contrast per-session (T3) with per-request dynamic evaluation (T4, covered next).

Concept primer

800-207 §2.1 T3 specifies that access is session-bound, not granted as an open-ended grant that can be replayed forever. Two session artefacts dominate practice: the human-session OAuth2/OIDC access token, bound to a subject (and optionally a key via DPoP or mTLS per the source §1 and §8.7), and the workload-session SPIFFE SVID, bound to a workload identity. The source document picks five-minute Keycloak/ForgeRock tokens and fifteen-minute SPIFFE SVIDs — short enough that theft is time-boxed, long enough that re-authentication cost is bearable. In this lab you set both to five minutes and watch them roll. T3 does not, by itself, guarantee that any given request inside a session is authorised (that is T4 and T6); it guarantees only that the session cannot outlive its intended lifetime, and that resumption requires a fresh proof. Crucially, "session" here is per-resource — a token good for the frontend is not good for the API unless its audience claim matches. That per-resource scoping is what the phrase "individual enterprise resources" in the tenet text insists on.

Step-by-step

01Bootstrap a Keycloak realm zta-bookstore with a 5-minute access-token lifespan, a confidential client bookstore-api, and a test user alice / alice.

Bashlabs/03-per-session/keycloak-realm.sh
#!/usr/bin/env bash
set -euo pipefail
CTX=docker-desktop
KC_POD=$(kubectl --context $CTX -n zta-identity get pod -l app=keycloak -o name | head -1)

kc() { kubectl --context $CTX -n zta-identity exec -i $KC_POD -- /opt/keycloak/bin/kcadm.sh "$@"; }

kc config credentials --server http://localhost:8080 --realm master --user admin --password admin

kc create realms -s realm=zta-bookstore -s enabled=true \
  -s accessTokenLifespan=300 \
  -s ssoSessionIdleTimeout=1800 \
  -s ssoSessionMaxLifespan=3600

kc create clients -r zta-bookstore -s clientId=bookstore-api \
  -s protocol=openid-connect -s publicClient=false \
  -s 'redirectUris=["http://bookstore.local/*"]' \
  -s directAccessGrantsEnabled=true \
  -s serviceAccountsEnabled=false
CID=$(kc get clients -r zta-bookstore -q clientId=bookstore-api --fields id --format csv --noquotes | tail -1)
kc update clients/$CID -r zta-bookstore -s 'attributes."access.token.lifespan"=300'
SECRET=$(kc get clients/$CID/client-secret -r zta-bookstore --fields value --format csv --noquotes | tail -1)

kc create users -r zta-bookstore -s username=alice -s enabled=true -s email=alice@zta.homelab
kc set-password -r zta-bookstore --username alice --new-password alice

echo "BOOKSTORE_CLIENT_SECRET=$SECRET" > labs/03-per-session/.env
echo "Realm ready. Secret stored in labs/03-per-session/.env"

04Deploy a thin sidecar-less watcher that uses the SPIFFE Workload API to fetch SVIDs and prints the serial number. Each rotation (every ~150 s, half the TTL) is visible.

YAMLlabs/03-per-session/svid-watcher.yaml
apiVersion: apps/v1
kind: Deployment
metadata: { name: svid-watcher, namespace: bookstore-api, labels: { zta.resource: "true", zta.data-class: "confidential", zta.owner: "retail-api", zta.tier-role: "api" } }
spec:
  replicas: 1
  selector: { matchLabels: { app: svid-watcher } }
  template:
    metadata:
      labels: { app: svid-watcher, zta.resource: "true", zta.data-class: "confidential", zta.owner: "retail-api", zta.tier-role: "api" }
    spec:
      serviceAccountName: default
      containers:
      - name: watcher
        image: ghcr.io/spiffe/spiffe-helper:0.9.0
        args: ["-config", "/config/helper.conf"]
        volumeMounts:
        - { name: config,   mountPath: /config }
        - { name: svid-out, mountPath: /svid }
        - { name: spire-agent-socket, mountPath: /run/spire/sockets }
      - name: observer
        image: alpine:3.20
        command: ["sh","-c","apk add --no-cache openssl >/dev/null; while true; do if [ -f /svid/svid.pem ]; then openssl x509 -in /svid/svid.pem -noout -serial -enddate -subject; fi; sleep 15; done"]
        volumeMounts:
        - { name: svid-out, mountPath: /svid, readOnly: true }
      volumes:
      - name: config
        configMap:
          name: svid-helper
      - { name: svid-out, emptyDir: {} }
      - name: spire-agent-socket
        hostPath: { path: /run/spire/sockets, type: DirectoryOrCreate }
---
apiVersion: v1
kind: ConfigMap
metadata: { name: svid-helper, namespace: bookstore-api }
data:

    agent_address = "/run/spire/sockets/agent.sock"
    cmd = ""
    cmd_args = ""
    cert_dir = "/svid"
    svid_file_name = "svid.pem"
    svid_key_file_name = "svid.key"

Regolabs/04-dynamic-policy/zta.authz.rego
package zta.authz

import rego.v1

# Envoy ext_authz input shape:
#   input.attributes.request.http.{method,path,headers}
#   input.attributes.source.principal          -- SPIFFE peer URI
#   input.parsed_path, input.parsed_query

default allow := false
default decision := {"allow": false, "reason": "default-deny"}

# --- helpers --------------------------------------------------------
token := t if {
  auth := input.attributes.request.http.headers.authorization
  startswith(auth, "Bearer ")
  t := substring(auth, 7, -1)
}

claims := c if {
  [_, payload, _] := io.jwt.decode(token)
  c := payload
}

posture := p if {
  p := input.attributes.request.http.headers["x-device-posture"]
} else := "unknown"

method := m if { m := input.attributes.request.http.method }
path   := p if { p := input.attributes.request.http.path }

workload_peer := s if { s := input.attributes.source.principal }

# --- rules ----------------------------------------------------------
# Allow: authenticated user + trusted posture + known workload peer
allow if {
  claims.sub
  posture == "trusted"
  startswith(workload_peer, "spiffe://cluster.local/ns/")
}

# Allow read-only GET from suspect devices
allow if {
  method == "GET"
  claims.sub
  posture == "suspect"
}

# Hard deny if device is tampered, regardless of token
decision := {"allow": false, "reason": "device-tampered"} if {
  posture == "tampered"
}

# Deny if unknown posture AND write method
decision := {"allow": false, "reason": "posture-unknown-on-write"} if {
  posture == "unknown"
  method != "GET"
}

# If a narrow deny rule didn't fire, project `allow` to decision
decision := {"allow": true,  "reason": "ok"}                    if allow
decision := {"allow": false, "reason": "no-matching-allow"}     if not allow

# Final response Envoy expects
result := {
  "allowed": decision.allow,
  "headers": {
    "x-zta-decision-id": decision_id,
    "x-zta-decision-reason": decision.reason,
  },
  "body": body,
  "http_status": status,
}

status := 200 if decision.allow
status := 403 if not decision.allow

body := "" if decision.allow
body := sprintf(`{"error":"forbidden","reason":"%s","decision_id":"%s"}`, [decision.reason, decision_id]) if not decision.allow

decision_id := crypto.sha256(sprintf("%v|%v|%v|%v|%v",
  [claims.sub, method, path, posture, time.now_ns()]))

Concept primer

Tenets 1–5 describe what is watched and how; Tenet 6 is the thesis statement — no request succeeds without a fresh, dynamic, authoritative authorisation, and that authorisation is enforced at the resource, not merely requested. The source §4 makes the policy plane Tier-0 for exactly this reason. In production the PA compiles Rego, runs opa test, signs the bundle with cosign, publishes it to an immutable bucket, and signs a manifest of trusted bundles the PE uses to reject anything else. In a home lab we shrink this to: a Deployment that runs opa build --signing-alg RS256 --signing-key key.pem, an HTTPS service that serves the resulting tarball, and an OPA configuration that includes the public key under keys and the bundle name under signing.keyid. The PEP side is already in place from Lab 4. The lab's value is seeing the loop close at request time: the same request, run twice, gets two different decisions because the posture input — signal, bundle, or policy — changed between calls.

#!/usr/bin/env bash
set -euo pipefail
source labs/03-per-session/.env

TOKEN=$(curl -s -H 'Host: keycloak.local' \
  -d 'grant_type=password' -d 'client_id=bookstore-api' \
  -d "client_secret=$BOOKSTORE_CLIENT_SECRET" \
  -d 'username=alice' -d 'password=alice' \
  http://localhost/realms/zta-bookstore/protocol/openid-connect/token \
  | jq -r .access_token)

echo "==> T3: token lifetime"
echo "$TOKEN" | cut -d. -f2 | base64 -d 2>/dev/null | jq '{exp, iat, lifetime_s: (.exp - .iat)}'

echo "==> T1-T2-T4-T6: denied round (suspect posture + POST)"
curl -sS -o /dev/null -w 'status=%{http_code}\n' \
  -X POST -H 'Host: bookstore.local' -H "Authorization: Bearer $TOKEN" \
  -H 'x-device-posture: suspect' \
  http://localhost/api/anything
# Expected: status=403

echo "==> T5: remediate — operator resets posture"
POD=$(kubectl --context docker-desktop -n bookstore-api get pod -l app=api -o name | head -1)
kubectl --context docker-desktop -n bookstore-api annotate --overwrite $POD zta.posture=trusted
kubectl --context docker-desktop -n bookstore-api delete $POD
kubectl --context docker-desktop -n bookstore-api rollout status deploy/api

echo "==> T6: allowed round"
curl -sS -o /dev/null -w 'status=%{http_code}\n' \
  -X POST -H 'Host: bookstore.local' -H "Authorization: Bearer $TOKEN" \
  -H 'x-device-posture: trusted' \
  http://localhost/api/anything
# Expected: status=200

echo "==> T7: the decision chain is visible"
kubectl --context docker-desktop -n zta-policy logs deploy/opa --tail=6 \
  | jq -c 'select(.decision_id) | {decision_id, allow: .result.allowed, reason: .result.headers["x-zta-decision-reason"]}' \
  | tail -2

Suggested extensions

  • Swap Keycloak for Dex — smaller footprint and pure-OIDC; loses admin UI but gains a tighter audit surface. Exercises the same source §7.1 role at lower complexity.
  • Add Cilium for L7 network policy (where kernel support allows). On Docker Desktop kernel feature support is limited; prefer running Cilium on a separate kind cluster with cluster-pool IPAM and enabling its Hubble UI. Adds a second §3.1.2 PEP below the mesh.
  • Second cluster for enclave simulation — stand up a kind cluster labelled swift-secure-zone, install a second Istio mesh with a mesh-federation trust bundle to the main cluster. Map directly to 800-207 §3.2.2 (enclave-based deployment) and source §5 Flow B.
  • Replace the Lua posture header with signed CDM events to Redis — removes the pod-bounce reconciler, shortens the T5→T6 loop to sub-second, and demonstrates a real-world upgrade path for source §7.14.
  • Cosign-signed OPA bundles with keyless Fulcio + Rekor — extends Lab 6 from RS256 private-key signing to sigstore's keyless flow (source §8.9). Requires network egress to the public Sigstore log; skip in air-gapped environments.
  • Add an AI Gateway PEP — a small egress proxy in front of an LLM endpoint that enforces prompt-injection filters and PAN/IBAN DLP, matching source §9.7. Uses the same OPA ext-authz pattern as Lab 4.
  • Turn zta.jurisdiction into a Gatekeeper mutation and have the Lab 4 Rego deny cross-jurisdiction writes — one clean demonstration of source §2 data-residency mediation.
  • Burn-rate alerts on SLOs — add Prometheus PrometheusRules for the §8.6 SLOs (decision latency P99 < 10 ms, availability 99.995%). Route to Alertmanager → stdout webhook for local development.
