Zero Trust Architecture on a Kubernetes Home Lab

By James Buckett · Product Manager

Deploy a working, NIST SP 800-207-aligned Zero Trust Architecture on your laptop. Seven labs take you from an empty Docker Desktop cluster to a signed-policy, fail-closed, telemetry-backed bookstore — each lab builds one 800-207 tenet, with runnable manifests, validation commands, and a break-it exercise. Written for Kubernetes practitioners who are new to Zero Trust.

By the end, you will have
  • A running single-node cluster with Istio, SPIRE, OPA, Gatekeeper, Falco, and Keycloak, wired into the NIST 800-207 PE/PA/PEP model
  • A Trust Algorithm that computes a trust_score and allows, step-ups, or denies requests against a sample banking API
  • A Grafana dashboard that traces every decision, keyed by decision_id, from ingress all the way to the data tier

1 · Executive Overview

The Seven Tenets of Zero Trust — your learning arc

Each lab targets exactly one tenet of NIST SP 800-207 §2.1. Read in order if you're new to Zero Trust; jump straight to a tenet if you already run OPA and Istio in production.

How to use this handbook. The seven labs build on each other — later labs assume the labels, mesh, and identity from earlier ones. If you skip ahead, the Capstone at the end will point you back to anything you haven't deployed. Each lab ends with a break-it exercise and a reflection block: do both. The dashboard you build in Lab 7 is what makes every earlier lab measurable.

The source document, ZTA-2026-001, is a C4 × NIST SP 800-207 architecture prepared by a global bank's Security Solutions Architecture team. In 800-207 terms it defines a distributed Policy Engine, a policy-as-code Policy Administrator, and a heterogeneous fleet of Policy Enforcement Points sitting in front of every resource. A Trust Algorithm pulls signals from CDM, SIEM, threat intel, UEBA, fraud/AML, and data-access policy to emit a continuous trust_score. This handbook reconstructs the essential shape of that architecture on a laptop — using only open-source projects — so that each 800-207 tenet becomes a thing you can deploy, break, observe, and repair.

2 · Lab Environment Setup

2.1 Prerequisites checklist

  • Docker Desktop 4.30+ (macOS, Windows with WSL2, or Linux) with built-in Kubernetes enabled. Settings → Kubernetes → Enable Kubernetes.
  • Resources: at minimum 6 CPU / 8 GB RAM / 60 GB disk. Istio + SPIRE + Falco + Keycloak + Grafana stack is chatty. 10 GB is more comfortable.
  • CLI tools: kubectl ≥ 1.29, helm ≥ 3.14, kustomize ≥ 5.0, istioctl ≥ 1.28, jq, yq, openssl.
  • Kubernetes context: docker-desktop. Every command in this tutorial is prefixed with --context docker-desktop or assumes you have set it as the current context.
  • Host entries for ingress: add 127.0.0.1 bookstore.local keycloak.local grafana.local to /etc/hosts (macOS/Linux) or C:\Windows\System32\drivers\etc\hosts (Windows).
Docker Desktop caveats
  • LoadBalancer: Docker Desktop's built-in Kubernetes assigns localhost as the external IP for Service type LoadBalancer — this is usable, but ports must be free.
  • Cilium: kernel-level features (eBPF bandwidth manager, L7 policies) are limited under Docker Desktop's Linux-in-VM kernel; we therefore use Istio as the primary mesh and mention Cilium only in the extensions.
  • Falco: we use the modern_ebpf driver, which works on Docker Desktop 4.30+ without kernel module compilation.

2.2 Install CLI utilities

Every lab in this tutorial drives Docker Desktop's Kubernetes through a small set of command-line tools. kubectl talks to the API server, helm renders the third-party charts (cert-manager, Falco, kube-prometheus-stack), kustomize patches the bookstore manifests, istioctl installs and introspects the mesh, jq / yq shape JSON and YAML on the pipe, and openssl inspects the X.509 and SVID material that Lab 2 will generate. Install all of them before touching the cluster — the bootstrap scripts fail fast if any one is missing.

kubectl
≥ 1.29 — Kubernetes API client; must match the server's minor version ± 1.
helm
≥ 3.14 — chart renderer for cert-manager, Falco, kube-prometheus-stack, Loki, Tempo.
kustomize
≥ 5.0 — overlay patcher for the bookstore workload manifests.
istioctl
≥ 1.28 — installs the mesh in §2.5 and introspects Envoy config in every lab. Istio 1.26 and earlier are EOL; 1.29 is the current upstream recommendation.
jq
≥ 1.7 — JSON slicer for kubectl get -o json and OPA decision logs.
yq
≥ 4.40 (Mike Farah) — YAML equivalent of jq; used by the scaffold and teardown scripts.
openssl
≥ 3.0 — inspects PeerAuthentication and SPIFFE SVID certificates in Lab 2.

  | sudo install /dev/stdin "${BIN}/kubectl"

curl -fsSL https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 \
  | HELM_INSTALL_DIR="${BIN}" bash

# kustomize — upstream installer; writes into the current directory, then moved.
curl -fsSL "https://raw.githubusercontent.com/kubernetes-sigs/kustomize/master/hack/install_kustomize.sh" \
  | bash
sudo mv ./kustomize "${BIN}/kustomize"

# istioctl — download, extract, promote.
ISTIO_VERSION="1.29.2"
curl -fsSL "https://github.com/istio/istio/releases/download/${ISTIO_VERSION}/istio-${ISTIO_VERSION}-linux-${ARCH}.tar.gz" \
  | tar -xz -C /tmp
sudo install "/tmp/istio-${ISTIO_VERSION}/bin/istioctl" "${BIN}/istioctl"

# yq — Mike Farah's Go implementation (NOT the Python yq).
YQ_VERSION="v4.44.3"
sudo curl -fsSL -o "${BIN}/yq" \
  "https://github.com/mikefarah/yq/releases/download/${YQ_VERSION}/yq_linux_${ARCH}"
sudo chmod +x "${BIN}/yq"

# Verify.
for b in kubectl helm kustomize istioctl jq yq openssl; do

Gotchas worth knowing before Lab 1
  • Two yq's exist. Ubuntu's apt install yq pulls the Python implementation, which uses a different syntax. Always install the Go version from mikefarah/yq — the scaffold and teardown scripts assume it.
  • kubectl skew. A client more than one minor version ahead of the server works for most operations but fails on newer resources (e.g. ValidatingAdmissionPolicy). Match the Docker Desktop Kubernetes minor where you can.
  • istioctl version ↔ control plane version. Lab 2 pins ISTIO_VERSION=1.29.2 (current upstream-supported minor; 1.26 and earlier are EOL). Your istioctl binary must be the same major/minor; upgrading the control plane with a mismatched CLI silently installs the CLI's version.
  • LibreSSL on macOS. The openssl shipped with macOS is LibreSSL and misses flags we use in Lab 2 (-verify_return_error, newer -subj parsing). Homebrew's openssl@3 fixes this — make sure $(brew --prefix openssl@3)/bin is earlier in PATH than /usr/bin.

2.5 Bootstrap — cert-manager, Istio, SPIRE

cert-manager provides a cluster-wide CA for non-SPIFFE TLS (Keycloak ingress, Grafana). Istio is the east-west PEP and sidecar mTLS enforcer. SPIRE issues short-lived X.509 and JWT SVIDs for workload identity — the 800-207 3.1.3 ID Management component in this lab.

Bashbootstrap/01-cert-manager.sh
#!/usr/bin/env bash
set -euo pipefail
CTX=docker-desktop

helm repo add jetstack https://charts.jetstack.io
helm repo update

helm --kube-context $CTX upgrade --install cert-manager jetstack/cert-manager \
  --namespace cert-manager \
  --version v1.15.3 \
  --set crds.enabled=true \
  --set global.leaderElection.namespace=cert-manager \
  --wait --timeout 5m

kubectl --context $CTX -n cert-manager wait --for=condition=Available \
  deploy/cert-manager deploy/cert-manager-webhook deploy/cert-manager-cainjector \
  --timeout=180s

# Cluster-wide self-signed issuer (non-SPIFFE paths only)
cat <<'EOF' | kubectl --context $CTX apply -f -
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: zta-selfsigned
spec:
  selfSigned: {}
EOF
echo "cert-manager ready."

Concept primer

The source document catalogues ~20 named components (7) and places them inside four logical layers (Data Sources, Control Plane, PEPs, Enterprise Resources) at C4 L2 (4). NIST 800-207 calls all of these — databases, APIs, services, even agents — resources, and Tenet 1 begins by insisting that the enterprise inventory them all and treat them as equally worthy of protection irrespective of whether they run on z/OS, EKS, or a laptop. In this lab the cluster plays the role of the bank's estate at miniature scale: the bookstore's frontend, api, and db are the "Subjects · Assets · Resources" box in the source diagram; the Istio sidecars are not resources, they are PEPs; OPA is the PE. The label schema you apply below becomes the primary key for every later lab's policy — policies will select resources by label, not by name, so that adding a new workload automatically inherits the correct enforcement without policy edits. This is the 800-207 3.4.1 Trust Algorithm input schema, rendered in Kubernetes-native form.

Two-pass apply is intentional

The file contains both a ConstraintTemplate and the K8sRequiredZtaLabels constraint it defines. On the first kubectl apply, the template is created — which causes Gatekeeper to register the K8sRequiredZtaLabels CRD. That registration is asynchronous, so the constraint in the same file fails with no matches for kind "K8sRequiredZtaLabels" ... ensure CRDs are installed first. We wait for the CRD to be Established, then re-apply so the constraint object lands against the now-known CRD. This is a known Gatekeeper admission-order quirk, not a lab error.

Concept primer

The source document's 3 Context warns that no network segment, including the corporate LAN, is trusted by default, and 4 places Istio at every east-west hop with SPIFFE SVIDs rotated every 60 minutes. The subtle point is that T2 does not by itself demand mTLS — it demands the strongest possible method — but in practice a modern mesh delivers mTLS cheaply enough that it is the default. What T2 rules out is the older defence-in-depth fallacy that perimeter firewalls can justify plaintext inside. The proof in this lab is deliberately adversarial: rather than reading the Istio control plane's claim that mTLS is on, you sniff a live packet from an untrusted-looking pod in an unrelated namespace. The bytes are opaque or not; there is no middle ground.

Concept primer

800-207 2.1 T3 specifies that access is session-bound, not granted as an open-ended grant that can be replayed forever. Two session artefacts dominate practice: the human-session OAuth2/OIDC access token, bound to a subject (and optionally a key via DPoP or mTLS per the source 1 and 8.7), and the workload-session SPIFFE SVID, bound to a workload identity. The source document picks five-minute Keycloak/ForgeRock tokens and fifteen-minute SPIFFE SVIDs — short enough that theft is time-boxed, long enough that re-authentication cost is bearable. In this lab you set both to five minutes and watch them roll. T3 does not, by itself, guarantee that any given request inside a session is authorised (that is T4 and T6); it guarantees only that the session cannot outlive its intended lifetime, and that resumption requires a fresh proof. Crucially, "session" here is per-resource — a token good for the frontend is not good for the API unless its audience claim matches. That per-resource scoping is what the phrase "individual enterprise resources" in the tenet text insists on.

01Author the Rego policy. Save it under the repo and load it into OPA as a ConfigMap.

Regolabs/04-dynamic-policy/zta.authz.rego
package zta.authz

import rego.v1

# Envoy ext_authz input shape:
#   input.attributes.request.http.{method,path,headers}
#   input.attributes.source.principal          -- SPIFFE peer URI
#   input.parsed_path, input.parsed_query

default allow := false
default decision := {"allow": false, "reason": "default-deny"}

# --- helpers --------------------------------------------------------
token := t if {
  auth := input.attributes.request.http.headers.authorization
  startswith(auth, "Bearer ")
  t := substring(auth, 7, -1)
}

claims := c if {
  [_, payload, _] := io.jwt.decode(token)
  c := payload
}

posture := p if {
  p := input.attributes.request.http.headers["x-device-posture"]
} else := "unknown"

method := m if { m := input.attributes.request.http.method }
path   := p if { p := input.attributes.request.http.path }

workload_peer := s if { s := input.attributes.source.principal }

# --- rules ----------------------------------------------------------
# Allow: authenticated user + trusted posture + known workload peer
allow if {
  claims.sub
  posture == "trusted"
  startswith(workload_peer, "spiffe://cluster.local/ns/")
}

# Allow read-only GET from suspect devices
allow if {
  method == "GET"
  claims.sub
  posture == "suspect"
}

# Hard deny if device is tampered, regardless of token
decision := {"allow": false, "reason": "device-tampered"} if {
  posture == "tampered"
}

# Deny if unknown posture AND write method
decision := {"allow": false, "reason": "posture-unknown-on-write"} if {
  posture == "unknown"
  method != "GET"
}

# If a narrow deny rule didn't fire, project `allow` to decision
decision := {"allow": true,  "reason": "ok"}                    if allow
decision := {"allow": false, "reason": "no-matching-allow"}     if not allow

# Final response Envoy expects
result := {
  "allowed": decision.allow,
  "headers": {
    "x-zta-decision-id": decision_id,
    "x-zta-decision-reason": decision.reason,
  },
  "body": body,
  "http_status": status,
}

status := 200 if decision.allow
status := 403 if not decision.allow

body := "" if decision.allow
body := sprintf(`{"error":"forbidden","reason":"%s","decision_id":"%s"}`, [decision.reason, decision_id]) if not decision.allow

decision_id := crypto.sha256(sprintf("%v|%v|%v|%v|%v",

Concept primer

T7 is often read as "collect lots of logs." The source 8.6 is more exact: SLOs are quantitative, telemetry feeds the Trust Algorithm, and observed allow-rate regressions are themselves an input to policy release gates. In this lab you hold up a magnifying glass to the cluster: Prometheus scrapes OPA's metrics endpoint for decision counters and latency histograms; the Loki Grafana Agent tails OPA pod logs, parses each line as JSON, and indexes on decision_id, reason, allow; Tempo receives OpenTelemetry traces from the Istio sidecars via the Zipkin endpoint. One Grafana dashboard stitches them together — a classic golden-signals view but with the decision itself as the first-class object. The improvement step is the point: you will notice that too many denies with reason no-matching-allow come from one API path, and you will tighten the Rego to give the operator a clearer error.

Glossary

A compact reference for the acronyms and concepts that appear across the seven labs. Grouped by role in the architecture, not alphabetically — the categories mirror how the pieces fit together in NIST SP 800-207.

NIST SP 800-207 logical components 7 terms

Policy Engine PE

The decision maker. Evaluates every request against policy and returns allow, deny, or step-up. In this lab: OPA with Rego policies reached through Envoy ext_authz.

NIST SP 800-207 §3.1.1
Policy Administrator PA

The distribution plane. Publishes signed, versioned policy bundles to every Policy Engine and revokes sessions. Home-lab analogue: a signed-bundle HTTP endpoint polled by OPA.

NIST SP 800-207 §3.1.1
Policy Enforcement Point PEP

The gate. Physically blocks or permits the traffic on behalf of the PE. Every hop has one — Istio sidecars, Kong for FAPI, ZPA for remote access, AAM for privileged sessions.

NIST SP 800-207 §3.1.2
Trust Algorithm TA

The logic inside the PE that weighs identity, posture, behaviour, and context. Starts criteria-based (hard rules) and evolves to score-based (weighted signals) as telemetry matures.

NIST SP 800-207 §3.3
Subject

The requester — a human user, a workload, or a service account. Identified through Keycloak (humans) or SPIFFE (workloads).

NIST SP 800-207 §2.2
Enterprise Resource

The protected asset — a service, dataset, API, or device. In the lab the demo bookstore exposes three tiers: frontend, api, and db.

NIST SP 800-207 §2.2
Data Sources

The facts the PE draws on at decision time: identity claims, CDM device hygiene, SIEM events, threat intel, UEBA drift scores, data-governance tags.

NIST SP 800-207 §3.1

Identity & workload attestation 7 terms

OpenID Connect OIDC

Identity layer over OAuth 2.0. Produces signed ID tokens that carry user claims. Keycloak is the OIDC provider for human identities in every lab.

Mutual TLS mTLS

Both client and server present X.509 certificates. In the mesh this is automatic — Istio rotates certs every 24 hours via cert-manager.

SPIFFE / SPIRE

The open standard for issuing workload identities. A SPIFFE ID like spiffe://home.lab/ns/bookstore/sa/api is embedded in a short-lived SVID certificate.

SVID

SPIFFE Verifiable Identity Document — the X.509 or JWT credential that proves a workload is who it claims to be. Rotates in minutes, not days.

WebAuthn / passkey

Phishing-resistant second factor. Private key is bound to the device; the server only stores the public key. Used for privileged sessions and break-glass in Labs 3 and 4.

DPoP

Demonstration of Proof-of-Possession. Binds an access token to a client key pair so stolen tokens are not replayable. Required by FAPI 2.0.

JWT

JSON Web Token — signed JSON payload. Tokens carry subject, audience, expiry, and a claim set that the PE evaluates.

Policy & enforcement 6 terms

Open Policy Agent OPA

General-purpose policy engine. Policies are written in Rego and distributed as signed bundles. Stand-in for the PE in every lab.

Rego

OPA's declarative policy language. Labs use Rego for request-level authorization (Lab 4) and trust-score aggregation (capstone).

Cedar

AWS-originated policy language with a stricter schema model. Called out in the source architecture as a Rego alternative for policy-as-data teams.

Gatekeeper

Kubernetes-native OPA admission controller. Blocks non-conforming manifests at apply time — the platform-layer PEP in Lab 6.

ext_authz

Envoy's external authorization filter. Every request pauses while Envoy asks the PE for a verdict. Sub-millisecond when OPA runs as a sidecar.

Attribute-Based Access Control ABAC

Access decided from attributes of the subject, resource, action, and environment — not from fixed roles. The architectural default in Lab 4.

Kubernetes & service mesh 5 terms

Namespace

Kubernetes' lightweight tenancy boundary. The lab uses one namespace per control-plane component plus bookstore for the workload.

Istio

Service mesh providing automatic mTLS, ingress/egress gateways, and the Envoy sidecars that serve as in-mesh PEPs.

Sidecar

A helper container colocated with the workload pod. Envoy sidecars intercept every byte in and out of the app — the per-pod PEP.

Admission Controller

Webhook the API server consults before persisting a manifest. Gatekeeper plugs in here to enforce posture, labels, and supply-chain provenance.

NetworkPolicy

Kubernetes' L3/L4 allow-list. Default-deny is assumed from Lab 1 onward; everything explicit flows through the mesh.

Telemetry & detection 6 terms

Continuous Diagnostics & Mitigation CDM

The running inventory of device and workload hygiene — patch level, config drift, agent health — fed back into the PE as posture input.

Security Information & Event Management SIEM

Aggregates and correlates security logs. Signals like "impossible travel" or "credential reuse" surface here and feed the TA.

User/Entity Behaviour Analytics UEBA

Baselines normal behaviour and emits a drift score when a subject deviates. One of the score-based inputs in Lab 4 and the capstone.

OpenTelemetry Protocol OTLP

Vendor-neutral wire format for traces, metrics, and logs. The mesh, OPA, and the app all emit OTLP to a single collector.

Falco

Runtime security monitor that reads syscalls and flags suspicious process, file, and network activity — the runtime detection layer in Lab 5.

Service Level Objective SLO

Measurable target for a quality attribute (e.g. decision latency P99 < 10 ms, 99.995% availability). Burn-rate alerts wake oncall when the budget is at risk.

Compliance frameworks 5 terms

NIST SP 800-207

The Zero Trust Architecture publication. Defines the seven tenets, the logical components, and the deployment variants. Every tenet in this lab anchors to its paragraph-level sections.

FAPI 2.0

OpenID Foundation's hardened OAuth profile — mandatory DPoP, PAR, mTLS, and strong client authentication. The external-API security baseline in the source doc.

Digital Operational Resilience Act DORA

EU regulation on ICT risk for financial institutions. Drives the telemetry retention, incident reporting, and supply-chain controls threaded through Labs 5–7.

Payment Card Industry DSS PCI DSS

Cardholder-data protection standard. Relevant to cryptographic controls (Lab 2), segmentation (Lab 6), and audit logging (Lab 7).

CISA Zero Trust Maturity Model ZTMM

CISA's five-pillar maturity framework — complements 800-207 by scoring where an organisation sits on each pillar from Traditional to Optimal.

Zero Trust concepts 6 terms

Implicit Trust Zone

A region of the network — a VLAN, VPN, or subnet — where traffic is assumed trusted because of where it came from. Zero Trust's defining move is to remove them.

Dynamic Policy

Policy whose verdict changes as context changes — time, device posture, behaviour, risk. The subject of Lab 4; contrasted with static ACL-style rules.

Step-up Authentication

Mid-session escalation — re-prompt for a passkey, a WebAuthn challenge, or an out-of-band approval — when the score drops into a caution band.

Least Privilege

Grant the minimum access needed, scoped to the session. Permissions expire with the session; they are not held forever.

Break-glass

The auditable, hard-to-fake path that allows urgent override — e.g. a two-person rule with WebAuthn. Described in Lab 6 and exercised in the capstone.

BeyondCorp

Google's foundational Zero Trust deployment — perimeter-less, device-and-user-aware, per-request authorized. A precursor to the SP 800-207 framing.
